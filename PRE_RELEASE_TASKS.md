# Актуальный статус перед релизом ArtStudio CMS

Документ синхронизирован с `main` 19 июля 2026 года. Старый список кодовых
блокеров закрыт; здесь остаются только действия, зависящие от конкретного
production-сервера и реального контента. Полный порядок запуска находится в
[`docs/RELEASE.md`](docs/RELEASE.md), технический деплой — в
[`docs/DEPLOY.md`](docs/DEPLOY.md).

## Закрыто в коде

- Прямой доступ к служебным PHP/SQL/config-файлам блокируется правилами
  Apache/nginx и CLI-guard; ручная проверка production URL описана в
  [`SECURITY.md`](SECURITY.md).
- Полный бэкап БД и загрузок, SHA-256, ротация «7 дневных + 4 недельных» и
  восстановление реализованы в `app/Console/backup_worker.php` и
  `database/restore.php`.
- Heartbeat воркеров, `/health`, статусы `failed`, ретраи и Telegram-алерты
  реализованы для фоновых очередей.
- Вложенное меню, библиотека из 37 типов блоков, Columns, варианты reveal,
  SVG-иконки, ревизии блоков и предпросмотр черновиков реализованы.
- CI выполняет четыре задачи: PHP 8.2–8.5 + MariaDB, PHPStan/Composer audit,
  JavaScript syntax и browser smoke Chromium. Текущий объём — 383 unit-теста
  без БД и 483 теста в полном прогоне.

## Обязательно перед открытием сайта

- [ ] Указать боевой `APP_URL`, `APP_ENV=production`, `APP_DEBUG=false`.
- [ ] Направить DocumentRoot на `public/` либо проверить актуальные `.htaccess`.
- [ ] Установить HTTPS и проверить редирект HTTP → HTTPS.
- [ ] Проверить, что служебные URL из `SECURITY.md` возвращают 403/404.
- [ ] Настроить cron всех воркеров и убедиться, что `/health` возвращает `ok`.
- [ ] Настроить внешнее хранилище бэкапов и выполнить тестовое восстановление.
- [ ] Настроить SMTP и Telegram, проверить вход с 2FA и доставку уведомлений.
- [ ] Заменить демо-контакты, изображения и материалы реальным контентом.
- [ ] Создать политику конфиденциальности и проверить формы согласия.
- [ ] Проверить sitemap, robots, canonical, Open Graph и редиректы старых URL.
- [ ] Включить OPcache, gzip/brotli и HTTP/2 на сервере.
- [ ] Добавить `/health` во внешний uptime-мониторинг.

## Проверки перед публикацией

```bash
php scripts/release_check.php
php tests/run.php
php scripts/smoke.php https://ваш-домен
```

После загрузки новой версии используйте fail-fast сценарий:

```bash
php scripts/release.php https://ваш-домен
```

Он проверяет окружение, создаёт бэкап, применяет миграции, обновляет шифрование
секретов, очищает кеш и запускает smoke. Доставку кода (`git pull`, архив или
панель хостинга) нужно выполнить отдельно до запуска сценария.

## После запуска

- [ ] Первую неделю контролировать `/health`, `storage/logs/error.log`, журнал
      действий и список недавних 404.
- [ ] Проверить фактическое восстановление последнего production-бэкапа.
- [ ] Замерить Lighthouse; целевой ориентир главной — Performance не ниже 90.
- [ ] Проверить очереди почты, соцсетей, Web Push и вебхуков на реальных
      интеграциях.
